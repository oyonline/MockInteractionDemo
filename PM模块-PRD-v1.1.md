## 1. 文档概述

### 1.1 目的
本文档定义了跨境电商中台系统中「新品开发PM模块」的完整产品需求，用于指导开发、测试及后续迭代。

### 1.2 适用范围
- 产品开发团队（项目经理、运营、市场、品牌、产品）
- 设计团队
- 供应链团队
- 系统开发团队

### 1.3 术语表

| 术语 | 说明 |
|------|------|
| SKU | Stock Keeping Unit，库存单位，指具体的产品规格 |
| 初评会 | 立项初评会议，决定是否进入打样阶段 |
| 终评会 | 产品终评会议，确定成本、预测，决定是否量产 |
| PM | Product Manager，产品经理/项目经理 |
| 老品运营 | 已上市产品的日常运营，包括补货、调拨、滞销处理 |

## 2. 模块概述

### 2.1 模块定位

**新品开发PM模块**是跨境电商中台系统的核心模块之一，负责管理从需求提出到产品上市的全生命周期，以及上市后的老品运营管理。

**解决的核心痛点**：
1. 不知道项目卡在哪一步
2. 有人拖进度但没提醒
3. 信息散落在各处，开会前要到处问
4. 借样品管理混乱

#### 3.2.1 新品开发7阶段定义

| 阶段 | 编码 | 准入条件 | 准出条件 | 核心产出 |
|------|------|---------|---------|---------|
| **01 需求调研** | 01 | 任何人提交需求单 | 需求单填写完整 | 调研报告（市场新品、竞品迭代） |
| **02 立项初评** | 02 | 需求单提交 | 初评会决议通过或跳过 | 初评会纪要(可选)，决策：是否值得立项 |
| **03 产品初评** | 03 | 初评会通过 | 第一轮报价完成 | 供应商对比、报价、产能信息，**可叫停** |
| **04 打样测试** | 04 | 初评会通过 | 样品确认OK | 手版样、测试报告、**可叫停** |
| **05 产品终评** | 05 | 样品确认OK | 终评会通过 | 成本锁定、销量预测（运营负责人，前6个月，分市场） |
| **06 首单开售** | 06 | 首单到仓 | 自动触发 | 开售标记 |
| **07 上市复盘** | 07 | 开售满2周/4周 | 报告生成 | 复盘报告（销量vs预测、退货率、ROI等） |

#### 3.2.2 终评后分叉流程

**产品终评通过后，两条线并行执行：**

```
                        产品终评通过
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
       【线1：供应链线】                【线2：素材线】
              │                               │
    采购 → 入国内仓                      文案策划
              │                               │
    质检 → 物流(报关/清关)               产品拍摄
              │                               │
    海运/空运 → 到港                    产品图/视频制作
              │                               │
    入海外仓/FBA仓 → 上架               链接装修
              │                               │
              │         【卡点】              │
              │    必须等实物样品到了才能拍    │
              │    （借样品问题头大）         │
              └───────────────┬───────────────┘
                              ▼
                    【汇合】正式开售首单
                              │
                    2周复盘 → 4周复盘
```

**线1 - 供应链线**:
- 采购 → 入国内仓 → 质检 → 物流(报关/清关) → 海运/空运 → 到港 → 入海外仓/FBA仓 → 上架

**线2 - 素材线**:
- 文案策划 → 产品拍摄 → 产品图 → 视频制作 → 链接装修 → 开售准备
- ⚠️ **关键卡点**: 必须等实物样品到了才能拍（借样品问题头大）

**汇合点**: 正式开售首单

### 3.3 老品运营流程

```
销量预测（多事业部综合规划）
        ↓
识别风险：
├── 滞销预警 → 打折等运营操作
├── 断货预警 → 是否停广告？
└── 断货补救 → 就近仓库调拨
        ↓
计划确认后：
┌─────────────┬─────────────┬─────────────┐
│  正常补货   │  特殊补货   │  滞销处理   │
├─────────────┼─────────────┼─────────────┤
│ 走完整供应  │ 仓间调拨    │ 打折等运营  │
│ 链流程      │ (从有库存的 │ 操作        │
│             │ 仓库调到缺货 │             │
│             │ 的业务仓)   │             │
└─────────────┴─────────────┴─────────────┘
```

#### 3.3.1 老品运营流程说明

**触发条件**: 已上市产品的日常运营管理

**参与者**: 多事业部协同（亚马逊/东南亚/国内）

**流程步骤**:
1. **销量预测**（多事业部综合规划）
   - 预测周期：滚动预测
   - 维度：SKU + 市场
   - 目的：统一采购及物流规划，合理降低成本

2. **风险识别**
   - **滞销预警**: 哪些品销售不及预期
   - **断货预警**: 哪些品可能断货
   - **断货补救**: 有没有就近的仓库能补，对应广告等投入是否要停

3. **执行决策**
   - **正常补货**: 走完整供应链流程（采购→入仓→质检→物流→上架）
   - **特殊补货**: 从某个仓调拨到对应缺货的业务仓
   - **滞销处理**: 打折等一系列运营操作，对应问题跟进处理

```
┌──────────────────┐         ┌──────────────────┐

│ current_stage    │         │ market           │
└────────┬─────────┘         │ forecast_v0      │ ←── 继承的初始预测
         │                   │ ...              │
         │                   └──────────────────┘
         │
         ▼
┌──────────────────┐
│  pm_project_sku  │
│  (项目SKU关联)    │
├──────────────────┤
│ id               │
```

**关联逻辑**：

2. **双向可追溯**：
   - 从老品运营可查看「来源新品项目」
   - 从新品项目可查看「转老品后的运营状态」
3. **多对一关系**：一个SKU可能经历多次新品开发（不同市场/版本），老品运营聚合展示

#### 3.4.4 预测数据继承策略

**继承规则**：

| 数据项 | 新品来源 | 老品继承 | 处理方式 |
|--------|---------|---------|---------|
| 预测周期 | 终评会（前6个月） | 滚动预测 | 作为v0基线，后续可覆盖 |
| 预测维度 | SKU+市场 | SKU+市场 | 保持一致 |
| 预测值 | 运营负责人录入 | 继承为初始值 | 标记为"新品预测" |
| 版本管理 | v1（终评） | v0（继承） | 新版本覆盖 |

**预测调整流程**：
```
首单开售（自动继承）
        │
        ▼
┌─────────────────┐
│   初始预测v0    │ ← 继承自新品终评会
│  （仅供参考）   │
└────────┬────────┘
         │
         │ 运营团队基于实际销售调整
         ▼
┌─────────────────┐
│   运营预测v1    │ ← 第一个正式运营预测
│  （生效版本）   │
└─────────────────┘
```

**关键说明**：
- v0预测仅作为参考基线，不参与老品运营的预警计算
- 运营团队需在首单开售后**7日内**提交v1预测
- v1预测未提交前，系统使用v0进行粗略预警（置信度低）

**自动触发的通知**：

| 接收人 | 通知内容 | 时间点 |
|--------|---------|--------|
| 运营负责人 | "SKU[XXX]已上市，请查看继承的预测数据并提交v1预测" | 首单开售确认后 |
| 供应链负责人 | "新品[XXX]已转老品运营，纳入正常补货规划" | 首单开售确认后 |

#### 3.4.6 异常处理

**场景1：首单开售延迟**
- 项目卡在"首单开售"阶段超过预期时间
- 系统预警："项目[XXX]首单延迟，请确认供应链状态"

**场景2：SKU信息变更**
- 新品阶段SKU编码变更
- 关联关系自动更新，老品运营同步变更

**场景3：多市场上市时间不一致**
- US市场先上市，UK市场延迟
- 先上市市场转老品运营，延迟市场继续新品流程
- SKU维度区分市场处理

### 3.6 素材优化任务流（独立触发）

**触发条件**: 正在销售的品感觉卖点/图片/视频不够吸引人

**流程**: 文案策划 → 卖点提炼 → 重新拍摄 → 制作 → 装修

**说明**:

- 可能随时触发

### 3.7 阶段流转规则

**流转触发条件**:
- 当前阶段任务全部完成
- 项目经理/管理员/节点负责人 可点击"推进至下一阶段"
- 系统记录推进人、推进时间

**跳过规则**:
- 立项初评会、产品终评会**可以跳过**（非强制）
- 跳过需填写跳过原因
- 跳过操作需管理员权限

**叫停规则**:
- 产品初评、打样测试阶段，如评估不通过，**可以叫停项目**
- 决策依据：预算花费、预期销量、成本
- 初评阶段投入较少，叫停成本低
- 打样阶段评估更准，叫停仍有价值

#### 3.7.1 任务完成判定标准

**"当前阶段任务全部完成"的判断逻辑**：

```
阶段完成判定
    │
    ├──→ 检查该阶段所有子任务状态
    │     ├── 全部子任务 status = 已完成 → 阶段可推进
    │     └── 任一子任务 status ≠ 已完成 → 阶段不可推进
    │
    └──→ 例外情况（人工干预）
          ├── 管理员可强制推进（需填写原因）
          └── 会议决议"跳过"当前阶段
```

**子任务完成判定（三选一）**：

| 判定方式 | 触发条件 | 适用场景 |
|---------|---------|---------|
| **自动完成** | 关联系统状态变更 | 供应链任务（ERP回传"已上架"） |
| **负责人标记** | 点击"完成任务"按钮 | 设计任务、文案任务 |
| **管理员确认** | 人工审核通过 | 测试报告、会议纪要 |

**任务状态流转**：
```
待开始 → 进行中 → 已完成
            ↓
         已暂停（项目叫停时）
```

#### 3.7.2 进度计算规则

**进度类型**：

| 进度类型 | 计算方式 | 说明 |
|---------|---------|------|
| **任务进度** | 负责人手动更新 | 0-100%，支持中间状态 |
| **阶段进度** | 自动计算 | 已完成任务数 / 总任务数 |
| **项目总进度** | 自动计算 | 加权平均各阶段进度 |

**任务进度（手动更新）**：

```
负责人更新方式：
├── 快速标记：0% / 25% / 50% / 75% / 100%
├── 精确输入：0-100 任意整数
└── 滑块拖动：可视化调整

更新要求：
- 必须填写"最新进展"文字说明
- 建议上传成果物截图/链接
- 系统自动记录更新时间和更新人
```

**阶段进度（自动计算）**：

```
阶段进度 = Σ(子任务权重 × 子任务进度) / Σ(子任务权重)

默认规则：
- 所有子任务权重相等 = 1
- 特殊任务可配置权重（如：测试报告权重=2）

简化显示：
- 阶段卡片显示：已完成任务数 / 总任务数
- 例：3/5 表示5个任务中已完成3个
```

**项目总进度（自动计算）**：

```
项目总进度 = Σ(阶段权重 × 阶段进度) / Σ(阶段权重)

阶段权重配置（默认）：
┌──────────┬────────┐
│ 阶段     │ 权重   │
├──────────┼────────┤
│ 需求调研 │ 10%    │
│ 立项初评 │ 10%    │
│ 打样测试 │ 20%    │
│ 产品终评 │ 15%    │
│ 并行执行 │ 25%    │
│ 首单开售 │ 10%    │
│ 上市复盘 │ 10%    │
└──────────┴────────┘
```

#### 3.7.3 阶段可推进性提示

**系统提示规则**：

| 场景 | 提示内容 | 按钮状态 |
|------|---------|---------|
| 有任务未完成 | "还有2个任务未完成，完成后可推进" | [推进按钮：灰色禁用] |
| 全部任务完成 | "当前阶段任务全部完成，可以推进" | [推进按钮：蓝色可用] |
| 强制推进 | "管理员可强制推进，需填写原因" | [强制推进：红色] |

**飞书提醒**：
- 任务截止前3天：提醒负责人"任务即将到期，请更新进度"
- 任务截止当天：提醒项目经理"有任务已延期"
- 阶段所有任务完成：提醒项目经理"当前阶段已完成，可推进"

### 3.8 异常流程与回退机制

#### 3.8.1 会议决议驳回处理

**场景**: 初评会/终评会决议为"驳回"

**处理规则**:

| 会议类型 | 驳回后状态 | 数据处理方式 | 后续操作 |
|---------|-----------|-------------|---------|
| **立项初评会** | 项目状态→"已驳回" | 保留需求单，清空阶段任务 | 可重新提交需求，生成新项目 |
| **产品终评会** | 回退至"打样测试"阶段 | 保留SKU数据，成本解锁 | 修改后重新提交终评 |

**数据保留策略**:
- 驳回后项目进入"已驳回"状态，不可再推进
- 历史记录完整保留，支持查看驳回原因
- 相关任务标记为"已取消"，保留执行记录

#### 3.8.2 项目叫停处理

**场景**: 初评/打样阶段评估不通过，决策"叫停"

**处理流程**:

```
项目叫停
    │
    ├──→ 项目状态变更为"已叫停"
    │
    ├──→ 冻结所有进行中的任务
    │     （状态：进行中→已暂停）
    │
    ├──→ 样品强制归还提醒
    │     ├── 在库样品：无操作
    │     ├── 借出样品：发送归还提醒给借用人
    │     └── 损耗样品：记录损耗原因
    │
    ├──→ 成本解锁（如有锁定）
    │
    └──→ 生成叫停报告（可选）
          （投入成本、已产生费用、叫停原因）
```

**叫停后状态管理**:

| 数据类型 | 处理方式 | 说明 |
|---------|---------|------|
| 项目基础信息 | 保留，状态=已叫停 | 不可编辑 |
| 阶段任务 | 保留，状态=已暂停 | 记录执行进度 |
| SKU数据 | 保留，成本解锁 | 恢复可编辑 |
| 样品记录 | 保留，触发归还提醒 | 实物在WMS系统处理 |
| 会议记录 | 保留 | 完整审计链 |
| 销量预测 | 保留 | 供复盘参考 |

**样品归还提醒**:
- 叫停后系统自动发送飞书消息给借用人
- 消息内容：项目[XXX]已叫停，请于X日内归还借用的样品
- 超期未归还：每日提醒一次，抄送项目经理

#### 3.8.3 阶段流转撤销

**场景**: 误操作推进到下一阶段

**撤销规则**:

| 条件 | 规则 |
|------|------|
| **可撤销窗口** | 推进后30分钟内 |
| **权限要求** | 仅推进人本人或管理员可撤销 |
| **撤销限制** | 下一阶段已有数据产生时不可撤销 |

**撤销流程**:
```
撤销申请
    │
    ├──→ 检查权限（是否本人/管理员）
    │
    ├──→ 检查时间窗口（30分钟内）
    │
    ├──→ 检查下一阶段数据状态
    │     ├── 无数据：允许撤销
    │     └── 有数据：提示"已有后续操作，无法撤销"
    │
    └──→ 执行撤销
          ├── 项目阶段回退
          ├── 记录撤销日志
          └── 通知相关人员
```

**不可撤销后的补救**:
- 如无法撤销，需走"阶段回退"流程（见3.8.4）

#### 3.8.4 阶段回退流程

**场景**: 需要回退到上一阶段（非误操作，业务需要）

**回退条件**:
- 需填写回退原因
- 需管理员权限
- 回退需经项目相关方确认

**回退影响**:

| 回退阶段 | 数据处理 |
|---------|---------|
| 终评→打样 | SKU成本解锁，预测数据保留历史版本 |
| 打样→初评 | 样品记录保留，测试任务标记"已取消" |
| 初评→调研 | 清空初评相关数据 |

**回退审批**:
- 提交回退申请 → 通知项目经理 → 确认后执行

#### 3.8.5 数据并发冲突处理

**场景**: 多人同时编辑同一项目/预测数据

**策略**: 乐观锁（Optimistic Locking）

**实现机制**:

```
数据表增加字段：
- version: int (版本号，默认1)
- updated_at: timestamp (最后更新时间)
- updated_by: varchar (最后更新人)
```

**冲突检测流程**:
```
用户A读取数据 (version=1)
    │
用户B读取数据 (version=1)
    │
用户A保存 (version=1 → 2, 成功)
    │
用户B保存 (version=1 ≠ 当前version=2)
    │
    └──→ 冲突！提示用户B：
          "数据已被其他人修改，请刷新后重试"
```

**冲突提示文案**:
> 该数据已被 **[updated_by]** 于 **[updated_at]** 修改，您的修改无法保存。
> 
> 请刷新页面查看最新数据后重试。

**批量编辑保护**:
- 批量更新操作时，检查所有目标记录的版本号
- 任一记录版本不匹配，整个操作回滚

**乐观锁 vs 悲观锁选择**:
- **乐观锁**：适用于读多写少的场景（PM模块主要场景）
- **悲观锁**：适用于高频并发写的场景（本模块不适用）

### 6.3 数据迁移方案

#### 6.3.1 历史数据现状

**现状**: 目前纯人工管理，数据散落在：
- Excel表格（项目跟踪表）
- 飞书文档（会议纪要）
- 飞书表格（样品借用）
- 各ERP系统（SKU基础信息）
- 邮件/聊天记录（沟通留痕）

#### 6.3.2 迁移策略

**策略**: "新项新系统，老项老办法" + 关键数据补录

```
系统上线时间点
    │
    ├──→ 【新项新系统】
    │     上线后新项目全部走PM系统
    │
    └──→ 【老项老办法】
          ├── 进行中的老项目：可选择性录入（关键里程碑）
          ├── 已完成的老项目：不录入，保留历史Excel
          └── 老品运营数据：从ERP系统同步基础SKU信息
```

#### 6.3.3 迁移范围

| 数据类型 | 迁移策略 | 优先级 | 说明 |
|---------|---------|--------|------|
| **进行中项目** | 选择性录入 | P1 | 只录入关键里程碑，不补全历史 |
| **SKU主数据** | 从ERP同步 | P0 | 自动同步，减少人工录入 |
| **样品台账** | 补录当前在库 | P1 | 只录当前状态，不补历史借还 |
| **已完成项目** | 不迁移 | - | 保留Excel，系统只管理新项 |
| **老品运营** | 从ERP同步 | P1 | SKU基础信息+当前库存 |

#### 6.3.4 并行期处理

**并行期**: 系统上线后 1-3个月

**双轨运行规则**:
- **新项目**: 强制走PM系统（P0功能必须上线才能切换）
- **老项目**: 可继续在Excel管理，或选择性录入系统
- **样品管理**: 系统记录为主，Excel台账为辅（1个月过渡期）

**切换检查点**:
| 检查项 | 标准 | 时间点 |
|--------|------|--------|
| 项目录入率 | 新项目100%录入 | 上线即生效 |
| 数据准确率 | 系统数据与实物一致 > 95% | 并行期结束 |
| 用户熟练度 | 核心用户培训通过 | 并行期第2周 |
| 系统稳定性 | 故障率 < 1% | 并行期第1个月 |

#### 6.3.5 数据补录工具

**Excel导入模板**:
- 提供标准导入模板（项目/SKU/样品）
- 支持批量导入
- 导入前数据校验（必填项、格式检查）
- 导入后生成报告（成功/失败明细）

**ERP同步接口**:
- 领星/速猫/旺店通 SKU基础信息同步
- 首次全量同步，后续增量同步
- 同步字段：SKU编码、产品名称、市场、售价

#### 6.3.6 回滚预案

**回滚触发条件**:
- 系统上线后1个月内，重大Bug无法快速修复
- 用户抵触严重，无法推动使用
- 数据丢失/错乱，影响业务

**回滚方案**:
- 保留Excel备份至少3个月
- 系统数据可导出为Excel
- 回滚后数据可人工整理重新导入
