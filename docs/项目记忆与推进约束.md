# 项目记忆与推进约束

> 更新时间：2026-04-01。本文档沉淀当前仓库的事实边界、已核实事实、禁止事项与模块推进原则。本文档中的“事实”与“推进约束”需分开理解，不可混用。

## 1. 已核实事实

### 1.1 仓库与技术形态

- 当前仓库是 React 19 + Create React App + Tailwind CSS 的前端仓库，语言为 JavaScript。
- 当前仓库应被视为“纯前端 + mock 数据的可交互原型仓库”，不是已进入真实后端联调阶段的完整业务系统。
- 当前仓库未使用 TypeScript、React Router、Redux/Zustand，也未发现现成的 HTTP 请求层依赖。
- 当前页面路由由 `src/App.js` 中的 `renderTabContent()` 和 `src/layouts/DynamicSidebar.js` 中的 `navConfig` 共同驱动。

### 1.2 当前数据来源现状

- `src/services/` 下已有 `settings.js`、`system.js`、`logistics.js`、`supply-chain.js`。
- 已核查到的 service 主要从 `src/mock/` 和 `localStorage` 读取/写入数据。
- 当前仓库的核心能力来自页面、mock、service 和 localStorage 的组合，不是来自真实接口、真实数据库或真实联调环境。
- 在 `src/` 内未检索到 `fetch`、`axios`、`request`、`graphql`、`apollo` 等真实请求实现。
- 部分页面走 `service -> mock/localStorage`，也有部分页面直接在页面文件中内嵌 mock 数据。
- `.gitignore` 预留了 `.env*` 本地文件忽略规则，但仓库内没有提交环境变量示例文件。

### 1.3 文档现状

- 在本次补充前，`docs/` 下只有 4 份文档，且没有独立的项目记忆类文档。
- 既有文档更偏“原型盘点”和“mock 方案描述”，缺少“当前事实边界 / 未来可选方向”的统一说明。

## 2. 当前推进约束

### 2.1 当前默认推进原则

- 当前默认定位是继续维护和完善前端原型仓库，不把未来产品化方向写成当前仓库事实。
- 优先最小改动、逐页闭环，不为了“看起来更整洁”先做大规模抽象或目录迁移。
- 在本仓可直接完成的任务，优先围绕页面、路由、菜单、mock、service、localStorage 展开。
- 如果后续获得明确外部后端信息，可优先沿现有页面/路由/菜单/service 链路逐页替换为真实接口，但这属于未来可选推进方向，不是当前仓库现状。

### 2.2 优先级

1. 先判断任务是否只涉及当前原型仓，还是必须依赖外部后端信息。
2. 如果任务只涉及当前原型仓，优先在页面、mock、service、localStorage 范围内完成。
3. 只有在外部后端信息明确后，才进入真实接口替换与联调语境。
4. 以页面为单位推进闭环；原型闭环与真实联调闭环要分开记录，不能混写。

### 2.3 禁止事项

- 不默认把当前 mock/service/localStorage 方案继续扩展为长期架构。
- 不在没有事实依据时设计全新前后端分层或大规模状态管理方案。
- 不通过大量 mock、假数据、假接口来掩盖真实联调缺口。
- 不顺手重构无关模块、批量改动页面结构或统一替换现有实现。
- 不把“已有页面/已有原型”表述成“已完成真实业务闭环”。
- 不把“未来可沿 service 替换真实接口”写成“当前已经具备真实接口能力”。

## 3. 模块推进原则

### 3.1 每次任务的默认顺序

1. 先读本文件和 `docs/文档索引.md`。
2. 再读 `docs/联调前置信息清单与缺口台账.md`，先区分仓库内已知与仓库外待补信息。
3. 再读目标模块相关文档、页面、service、mock。
4. 如任务涉及未来真实接口替换，再确认是否已有真实后端能力、接口字段、数据库依据。
5. 只对当前页面闭环所需的最小代码做改动。
6. 完成后补文档，记录事实变化、未解问题和下一步建议。

### 3.2 “逐页闭环”的判定标准

- 当前页的核心数据来源已明确。
- 如果任务明确进入真实接口替换，页面关键交互应尽量走真实接口；否则继续沿当前原型链路推进并清楚标注边界。
- 相关列表/详情/编辑的跳转与保存链路可验证。
- 遗留问题有明确记录，而不是被临时假数据掩盖。

### 3.3 对现有 mock 的处理原则

- 现有 mock 可以作为字段、交互、页面结构参考。
- 如果要保留 mock，只能作为临时兜底，并且要清楚标注替换条件。
- 在当前仓库阶段，新任务既可以是原型完善，也可以是为未来替换真实接口做替换点梳理；两者都不能被误写成“已真实联调”。

## 4. 子 Agent 上手清单

- 先看 `README.md`，确认仓库定位和入口。
- 再看 `docs/文档索引.md`，判断哪份文档是事实、哪份是约束、哪份只是历史快照。
- 再看 `docs/联调前置信息清单与缺口台账.md`，确认哪些联调信息仓库内已知、哪些必须向外部补。
- 接着看 `docs/项目架构与开发指南.md`，确认当前目录、组件复用、路由接入方式。
- 最后再进入目标模块的页面文件、service 文件和相关专题文档。

## 5. 仍缺失的信息

- 后端仓库位置、服务地址、接口鉴权方式。
- 真实数据库结构、核心表关系、字段口径。
- 各模块的接口清单、页面与接口映射、真实联调完成度。
- 联调环境的启动方式、测试账号、基础数据准备方法。

## 5.1 当前最适合继续推进的任务

- 页面原型完善、文案与交互补齐。
- mock、service、localStorage 现状梳理与替换点标注。
- 路由、菜单、页面跳转链路补齐。
- 文档纠偏，明确“当前事实”和“未来方向”的边界。

## 5.2 现在不应误开的任务

- 把当前仓库表述成已进入真实后端联调阶段。
- 在没有外部后端资料时开始真实接口接入、真实数据库映射或联调执行。
- 把未来产品化路径写成当前已具备能力。

## 6. 维护约定

- 当仓库事实发生变化时，优先更新“事实”部分。
- 当团队执行策略发生变化时，更新“推进约束”部分。
- 不把一次性的任务说明直接沉淀成长期约束。
