# 文档索引

> 更新时间：2026-04-01。本文档用于回答两个问题：当前该看哪份文档、每份文档应该被当作“事实”“约束”还是“历史材料”来使用。阅读时请先以“本仓库是纯前端 + mock 数据的可交互原型”作为事实前提。

## 推荐阅读顺序

1. `README.md`
2. `docs/项目记忆与推进约束.md`
3. `docs/联调前置信息清单与缺口台账.md`
4. `docs/项目架构与开发指南.md`
5. 与目标模块直接相关的专题文档

## 当前文档职责盘点

| 文档 | 主要角色 | 信息属性 | 当前使用建议 |
|------|----------|----------|--------------|
| `README.md` | 仓库总览、快速启动、入口说明 | 事实 + 入口 | 第一次进入仓库先看，重点看仓库定位与文档入口 |
| `docs/项目记忆与推进约束.md` | 当前仓库事实边界、禁止事项、默认推进原则、任务边界 | 事实 + 推进约束 | 后续子 Agent 开始工作前必读，优先级高于历史方案类文档 |
| `docs/联调前置信息清单与缺口台账.md` | 仅在需要真实接口替换时使用的前置清单、仓库内已知事实、外部待补缺口 | 事实 + 缺口台账 | 只有在任务明确进入真实接口替换前，才需要重点阅读 |
| `docs/项目架构与开发指南.md` | 前端代码组织、页面接入方式、样式与组件约定 | 事实（结构/规范） | 需要改页面结构、接路由、复用组件时查看 |

| `docs/prototype/项目现状盘点报告-物流与报关.md` | 物流与报关模块的原型实现盘点 | 模块快照 | 仅用于了解该模块已有页面与 mock/service 组织，不等于已完成真实联调 |
| `docs/mock-service-storage-方案.md` | 当前 mock/service/storage 方案说明 | 历史方案 + 现状描述 | 用于理解现有前端数据流，不能作为后续默认扩展方向 |

## 重叠点

- `README.md` 与 `docs/项目架构与开发指南.md` 都在介绍项目定位和目录结构。
- `docs/prototype/前端原型与mock能力盘点.md` 与 `docs/mock-service-storage-方案.md` 都在描述 mock、service、localStorage。
- `docs/项目架构与开发指南.md` 与 `docs/prototype/项目现状盘点报告-物流与报关.md` 都包含“模块状态”表达，但粒度和可信边界不同。

## 过时点与使用风险

- `docs/prototype/前端原型与mock能力盘点.md` 中部分示例路径仍沿用旧页面路径命名，例如 `SupplierListPage.js` 的旧引用方式，使用时要以 `src/pages/` 当前目录结构为准。
- `docs/prototype/项目现状盘点报告-物流与报关.md` 将“页面存在/可用”与“真实接口闭环”放在同一语境中，容易被误读为已完成真实联调。
- `docs/mock-service-storage-方案.md` 使用了“已完全实施”的措辞，但它描述的是当前前端 mock 方案已落地，不代表这就是后续主线。
- 旧文档普遍缺少“事实”和“未来方向”的明确分层，容易把“未来可产品化/可联调”误写成“当前已经具备真实后端能力”。

## 当前缺口

- 缺少后端能力盘点、接口清单、数据库表映射、环境联调方式等真实联调资料。
- 缺少按模块记录“哪些页面已真实联调、哪些仍是 mock”的持续更新台账。
- 隐藏在 `.cursor/` 下的原型规则可作为历史线索，但部分描述已经过时，不能直接视为当前事实文档。

## 后续补充建议

- 当外部后端信息明确后，可新增“后端能力与接口映射”文档，按模块记录真实接口、鉴权、字段映射和依赖环境。
- 当外部联调真正开始后，可新增“逐页联调进度”文档，明确每一页的完成定义、阻塞点和验证方式。
- 在当前阶段，更适合继续补充“原型事实边界”和“页面/服务替换点”说明，而不是把未来产品化方案写成当前能力。
